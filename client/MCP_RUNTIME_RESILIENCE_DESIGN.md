# MCP Runtime 稳定性增强设计

## 目标

在现有 MCP stdio 运行时基础上增加健康检查、异常断开自动重连、启动超时治理、可靠进程清理和有限调用恢复，同时保留手动启停与工具审批的安全边界。

## 状态与指标

运行状态扩展为：

- `stopped`：用户主动停止或尚未启动。
- `starting`：正在建立连接并发现工具。
- `running`：连接正常且工具已注册。
- `reconnecting`：异常断开后等待或正在重连。
- `error`：启动失败或自动重连耗尽。

公开状态增加：

- `lastHealthCheckAt`：最近一次成功健康检查时间。
- `consecutiveFailures`：连续启动、健康检查或连接失败次数。
- `reconnectAttempt`：当前重连序列的尝试次数。
- `nextReconnectAt`：下一次自动重连时间。
- `lastDisconnectedAt`：最近异常断开时间。

## 健康检查

运行时每 30 秒调用 MCP `ping`。成功后更新健康时间并清零连续失败；失败时先记录错误，连续两次失败后关闭旧连接并进入自动重连。健康检查按 Server 串行执行，不与启停操作并发。

提供 `POST /api/mcp/servers/:id/health` 手动探测接口。手动探测不改变用户的 autoStart 配置，也不启动已经停止的 Server。

## 自动重连

仅在以下条件同时成立时重连：

- 连接不是用户主动关闭；
- Server 配置仍存在、启用且 `autoStart=true`；
- 应用没有进入整体关闭流程。

退避序列为 1、2、4、8、16、30 秒，最多尝试 6 次，并加入少量随机抖动。连接成功后清零重连计数。用户执行 stop、删除或修改配置时必须取消定时器；用户手动 start/restart 时立即重置退避序列。

## 启动超时与清理

- connect 和工具发现共享 20 秒启动预算。
- 超时或发现失败时注销已经注册的工具、关闭 Client/transport，并清理 active map。
- shutdown 时取消全部健康检查与重连定时器，再并行关闭活动连接。
- `onclose` 必须验证关闭事件仍属于当前 active 实例，防止旧进程退出覆盖新连接状态。

## 工具调用恢复

默认不自动重试 `callTool`，因为工具可能产生写入副作用，超时不代表远端未执行。运行时只在调用开始前发现连接已经不可用时触发重连提示，不重放请求。

对于明确只读的 MCP 工具，当前 schema 没有可靠的幂等元数据，因此本阶段仍不自动重试。后续可增加显式 `idempotent/readOnly` 配置后，再允许一次连接级重试。

## API 与 UI

- Server 列表展示 `reconnecting`、重连次数、下次重连时间、最近健康时间和连续失败数。
- 增加“健康检查”操作。
- error/reconnecting 状态保留手动 start/restart/stop 控制。
- 后端返回结构化状态，前端不自行推断连接健康。

## 测试

- 正常启动后健康检查更新时间。
- 用户 stop 不触发自动重连。
- 异常断开按退避重连并最终恢复工具注册。
- 旧连接迟到的 close 事件不影响新连接。
- 自动重连耗尽后进入 error。
- shutdown 清除 timer、工具注册和活动连接。
- 工具调用超时不自动重放。
