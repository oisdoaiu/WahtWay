# WahtWay 文件操作安全规范

> 参考：Claude Code Permission Model · OpenAI Code Interpreter · Cursor/Windsurf GhostApproval 漏洞报告
> 用途：开发文件写操作 Tool 时的检查清单

---

## 铁律（每条都必须遵守）

### 1. 写操作必须用户确认

- 读文件（list-files、read-file、search-files、file-info）→ 不确认
- 写文件（移动、删除、重命名、复制、新建）→ **必须弹确认框**
- 没有任何写操作可以跳过用户确认

### 2. 删除 = 回收站，不是永久删除

- **禁止** `fs.unlinkSync` 用于用户文件
- 删除操作 = 移动到 `.wahtway-trash/` 目录
- 用户可以从回收站恢复

### 3. 确认框必须展示完整信息

```
❌ 坏："确认删除这个文件？"
✅ 好：
   操作: 移动文件
   源:   C:\Users\asus\Downloads\report.docx
   目标: C:\Users\asus\Documents\报告\report.docx
```

必须展示：操作类型、源路径、目标路径（如有）。

### 4. 先确认再动，不能先动再问

- Windsurf 的致命 bug：文件写盘了，确认框才出来
- WahtWay：Tool execute 分两步——
  1. 后端返回操作摘要 → SSE 推 `tool_confirm` 事件
  2. 前端弹确认框 → 用户点确认 → 后端真正执行

### 5. 一次只动一个文件

- 串行操作，不批处理
- LLM 一次调一个 Tool，agenticLoop 自然保证串行

### 6. 路径安全

| 检查项 | 规则 |
|------|------|
| 白名单 | 只能操作 `C:\Users\<用户名>\` 下的文件 |
| 系统目录 | `C:\Windows`、`C:\Program Files`、`C:\ProgramData` 直接拒绝 |
| 路径解析 | 先 `fs.realpathSync` 解析 symlink 和 `..`，再判断 |
| 隐藏目录 | `.git` 内文件禁止写入 |

## Tool 分类与确认策略

| Tool | 分类 | 确认 | 备注 |
|------|:---:|:---:|------|
| `list-files` | 读 | ❌ | 不确认 |
| `read-file` | 读 | ❌ | 不确认 |
| `search-files` | 读 | ❌ | 不确认 |
| `file-info` | 读 | ❌ | 不确认 |
| `move-file` | 写 | ✅ | 显示源/目标 |
| `copy-file` | 写 | ⚠️ | 可选确认（低危） |
| `delete-file` | 写 | ✅ | 移到回收站 |
| `rename-file` | 写 | ✅ | 同 move-file |
| `organize-files` | 写 | ✅ | 多文件，显示摘要 |
| `new-folder` | 写 | ❌ | 低危，不确认 |
| `write-file` | 写 | ✅ | 创建/覆写文件 |

## 确认框 UI 规范

### 弹框模板

```
┌─────────────────────────────────────────┐
│  📋 操作确认                             │
│                                         │
│  类型: 移动文件                          │
│  源路径: C:\Users\asus\Downloads\a.txt  │
│  目标路径: C:\Users\asus\Desktop\a.txt  │
│                                         │
│  ⚠️ 目标已存在，将被覆盖                  │  ← 仅在覆盖时显示
│                                         │
│         [取消]    [确认移动]              │
└─────────────────────────────────────────┘
```

### 危险标记

- 目标路径在用户目录外 → 🔴 红色警告
- 操作涉及超过 10 个文件 → 🟡 "将影响 N 个文件，确认？"
- 源路径包含系统文件后缀（`.dll`, `.sys`）→ 🔴 拒绝

## 回收站设计

```
C:\Users\<用户名>\.wahtway-trash\
├── 2026-07-16_143052_report.docx     ← 删除的文件
├── 2026-07-16_143105_photo.png
└── trash-log.json                     ← 记录原始路径和删除时间
```

### trash-log.json 格式

```json
[
  {
    "originalPath": "C:\\Users\\asus\\Desktop\\old.txt",
    "trashName": "2026-07-16_143052_old.txt",
    "deletedAt": "2026-07-16T14:30:52",
    "size": 1024
  }
]
```

## 禁止操作清单

| 操作 | 处理方式 |
|------|------|
| 删除系统目录下的文件 | 🔴 直接拒绝 |
| 移动文件到系统目录 | 🔴 直接拒绝 |
| 操作 `.env`、`id_rsa`、`authorized_keys` | 🔴 直接拒绝 |
| 批量删除超过 50 个文件 | 🟡 要求用户确认数量 |
| 递归删除目录 | 🔴 拒绝，要求逐个文件操作 |

## 实现检查清单

开发写操作 Tool 时逐条核对：

- [ ] 是否先确认再执行？
- [ ] 确认框是否展示了源路径和目标路径？
- [ ] 删除操作是否移到回收站而非永久删除？
- [ ] 是否检查了路径白名单（用户目录内）？
- [ ] 是否拒绝了系统目录操作？
- [ ] 是否一次只操作一个文件？
- [ ] 是否拒绝了敏感文件操作（.env、密钥等）？
